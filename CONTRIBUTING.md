# Contributing to hello-github

Thank you for your interest in contributing to hello-github! This document provides guidelines and instructions for contributing to this project.

## 🎯 Purpose

This repository is a learning resource for GitHub workflows and Git commands. Contributions that improve the documentation, add practical examples, or enhance the learning materials are highly valued.

## 🚀 Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/your-username/hello-github.git`
3. Create a new branch for your contribution
4. Make your changes
5. Test your changes
6. Submit a pull request

## 📝 Contribution Guidelines

### Types of Contributions

| Type | Description |
|------|-------------|
| **Documentation** | Improve existing docs, fix typos, add explanations |
| **Examples** | Add practical code examples for Git/GitHub features |
| **Tutorials** | Create step-by-step guides for common workflows |
| **Translations** | Help translate documentation to other languages |
| **Suggestions** | Propose new topics or improvements via issues |

### What We Look For

- Clear, concise, and accurate content
- Practical, real-world examples
- Well-formatted markdown
- No hardcoded secrets or sensitive data
- Follows the project's structure and style

## 🤝 How to Contribute

### Pull Request Process

1. **Create a branch** from `main` with a descriptive name:
   ```bash
   git checkout -b feat/improve-merge-conflict-guide
   # or
   git checkout -b fix/typo-in-readme
   ```

2. **Make your changes** following the code style below

3. **Test your changes** - ensure no broken links or formatting issues

4. **Commit your changes** using conventional commits:
   ```bash
   git commit -m "feat: add example for GitHub Actions"
   git commit -m "fix: correct merge command in README"
   ```

5. **Push to your fork**:
   ```bash
   git push origin feat/improve-merge-conflict-guide
   ```

6. **Open a Pull Request** from your branch to `main`

### PR Title Format

Use conventional commits format:
- `feat:` - New feature
- `fix:` - Bug fix
- `docs:` - Documentation changes
- `refactor:` - Code refactoring
- `perf:` - Performance improvements
- `test:` - Adding or updating tests

## 💻 Code Style

### Markdown Style

- Use proper markdown formatting
- Keep lines under 100 characters
- Use consistent heading hierarchy
- Include code blocks with language identifiers:

  ```markdown
  ```bash
  git status
  ```
  ```

### Formatting

- 2 spaces for indentation
- No trailing whitespace
- Single newline at end of file
- Use [Emojis](https://github.com/ikatyang/emoji-cheat-sheet) sparingly for visual organization

## 🐛 Reporting Issues

When reporting issues, include:

1. **Clear description** of the problem
2. **Steps to reproduce** the issue
3. **Expected behavior** vs **actual behavior**
4. **Environment information** (OS, browser, etc.)
5. **Screenshots** if applicable
